# Conventional Knowledge Commits v0.1.0

## Summary

Conventional Knowledge Commits (CKC) is a lightweight convention for commit messages that record
**knowledge work** — mathematical claims and proofs, and empirical scientific findings — on top of
[Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/).

CKC is a **strict superset**: every CKC commit is a valid Conventional Commit. It adds a research
vocabulary (commit *types*), an *epistemic-status* discipline, and structured *footers* (git
trailers) that together let tooling build a **knowledge dependency graph** and track, honestly,
what is *conjectured* vs *proved* vs *axiomatised* vs *refuted* — and what a refutation or a closed
gap implies for everything that depended on it.

A CKC commit captures a claim, its status, and what it depends on.

```
<type>[(scope)][~][!]: <description>

[body]

[footers]
```

## The two axes (the core idea)

Software change has **one** lattice: backward-compatibility, which Conventional Commits maps onto
SemVer (`fix`→PATCH, `feat`→MINOR, `BREAKING CHANGE`→MAJOR). Knowledge change has **two
orthogonal** axes, and separating them is the whole design:

- **Axis 1 — dependency impact** *(immutable; lives in the `type` and `!`)*. Does this commit
  invalidate things built on it? This is the genuine SemVer analog: a *repair* (PATCH-like), an
  *additive* result (MINOR-like), or a *breaking* change (MAJOR-like) such as refuting, retracting,
  re-stating, or redefining something others depend on.
- **Axis 2 — epistemic status** *(mutable; lives in the `Status:` footer)*. How strongly is the
  claim established right now? `conjectured → proved → machine-checked` (math), or
  `hypothesis → measured → replicated` (science). Status is advanced by **later commits**, never by
  rewriting an earlier one. Closing a `sorry` is a new `formalize` commit that flips
  `Status: math.open → math.machine-checked` — the original history stands.

Keeping these apart is why CKC does not need to rewrite a commit's type when a proof later closes,
and why a single `#print axioms` result can be both an Axis-2 status and an Axis-1 caveat.

## Profiles

CKC defines one core and two **profiles**:

- the **`proof`** profile — mathematics and formal proving (Lean/Mathlib-coupled);
- the **`science`** profile — empirical discovery (experiments, data, replication).

**The type implies the profile.** Shared types are profile-neutral; profile-specific types belong
to exactly one profile. A `Profile:` footer is used only to disambiguate a genuinely cross-profile
commit. Per-profile detail lives in [`proof-profile.md`](./proof-profile.md) and
[`science-profile.md`](./science-profile.md).

## Specification

The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHALL”, “SHALL NOT”, “SHOULD”, “SHOULD NOT”,
“RECOMMENDED”, “MAY”, and “OPTIONAL” are to be interpreted as described in
[RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

1. Every CKC commit **MUST** be a valid Conventional Commit 1.0.0 message: a `type`, an OPTIONAL
   `scope`, an OPTIONAL `!`, a REQUIRED terminal colon and space, and a `description`; with an
   OPTIONAL body and OPTIONAL footers, separated by blank lines.

2. The `type` **MUST** be a noun. It SHOULD be one of the CKC types — shared
   (`conjecture`, `lit`, `review`, `refute`, `retract`, `expose`, `meta`), `proof`-profile
   (`state`, `proof`, `formalize`, `axiomatize`, `strengthen`, `generalize`, `weaken`, `port`), or
   `science`-profile (`experiment`, `result`, `replicate`, `null`, `data`, `protocol`, `method`,
   `analysis`, `repro-fix`) — or a Conventional-Commits type (`feat`, `fix`, `chore`, `build`, `ci`,
   `docs`, `refactor`, `test`) for surrounding tooling and prose.

3. A `scope` MAY be provided. For knowledge commits it SHOULD name the **claim or area** the commit
   concerns, e.g. `formalize(fubini):`.

4. **Axis 1 (dependency impact).** A commit that invalidates results which depend on it — refuting,
   retracting, re-stating a theorem so dependents no longer hold, redefining a term, or a data
   correction that changes downstream conclusions — **MUST** signal it, either with a `!` immediately
   before the `:`, or with a `BREAKING CHANGE:` footer (synonym `BREAKING-CHANGE:`), exactly as in
   Conventional Commits. A `BREAKING CHANGE:` footer, when present, **MUST** name the affected
   claim(s). A commit MAY additionally make Axis 1 explicit with an `Impact: patch | minor | major`
   footer.

5. **Axis 2 (epistemic status).** A commit that asserts or changes the status of a claim SHOULD
   carry a `Status:` footer with a namespaced value (`math.*` or `sci.*`, see the profiles). Status
   **MUST NOT** be changed by amending or rewriting an earlier commit; it is advanced (or demoted)
   by a **new** commit referencing the same claim.

6. **Trusted-base markers.** A commit whose results rest on a newly introduced assumption **MUST**
   declare it with a loud, uppercase footer (mirroring `BREAKING CHANGE`): in the `proof` profile,
   `AXIOM:` (a cited/added axiom) or `OPEN:` (a `sorry`/`admit`/`sorryAx`); in the `science`
   profile, `ASSUMES:` (an untested assumption) or `UNREPLICATED:` (a single, un-replicated run). A
   trusted-base marker affects **Axis 2** (the status becomes `axiomatised`/`open`/`unreplicated`);
   it is **Axis 1 breaking only if** it also invalidates an existing dependent.

7. **Stable identity.** A claim, theorem, definition, or hypothesis that other commits reference
   **MUST** have a stable identifier (see [`identifiers.md`](../identifiers.md)). In the `proof`
   profile this is the Lean fully-qualified name (`Lean:` / `Formal-Statement: lean:<FQN>`) and/or a
   blueprint label; in the `science` profile a curated slug registered in the claims registry. Commits
   **SHOULD NOT** silently rewrite a claim under an unchanged identifier — use `Supersedes:` or an
   Axis-1 breaking change.

8. **Relations.** Footers expressing relations between claims **MUST** use the git-trailer form
   (`Token: value`, the token using `-` for spaces) and MAY be repeated. Defined relation tokens:
   `Depends-On`, `Assumes`, `Proves`, `Refutes`, `Disproves`, `Supersedes`, `Retracts`,
   `Invalidates`, `Closes`. These edges define the knowledge graph (see
   [`impact-graph.md`](../impact-graph.md)).

9. **Provenance.** Verification and provenance footers SHOULD be recorded where they apply:
   `Lean`, `Formal-Statement`, `Axioms`, `Verified-By`, `Proof-Hash`, `Cites`, `Phase`, `Gate`
   (proof); `Dataset`, `Protocol`, `Method`, `Sample-Size`, `Effect-Size`, `CI`, `P-Value`, `Seed`,
   `Hardware`, `Metric`, `Pre-Registration` (science). The `Axioms:` footer, when present for a
   formalized result, SHOULD be the literal output of `#print axioms`.

10. **Negative results are first-class.** Counterexamples, disproofs, null results, and failed
    replications **MUST** be expressible as ordinary commits (`refute`, `null`, a `replicate` with a
    `sci.not-replicated` status) — never hidden, deferred, or downgraded to a `fix`.

11. **Granularity.** A commit SHOULD concern **one** claim, theorem, or finding, so that the
    knowledge graph has well-defined nodes. Bundle only genuinely inseparable changes.

12. **Optional `~` sigil.** A `~` MAY be placed immediately before the `:` (after any `scope`/`!`)
    as an at-a-glance marker that the commit carries a trusted-base caveat (an `AXIOM:`/`OPEN:`/
    `ASSUMES:`/`UNREPLICATED:` footer). The footer remains the canonical, machine-readable record;
    the `~` is human sugar and MUST NOT be the sole indicator.

13. The units of information that make up CKC **MUST NOT** be treated as case-sensitive by
    implementors, with the exception of the uppercase trusted-base/breaking tokens
    (`BREAKING CHANGE`, `AXIOM`, `OPEN`, `ASSUMES`, `UNREPLICATED`), which MUST be uppercase.

## Relation to SemVer and the payoff

The primary downstream artifact CKC serves is the **knowledge dependency + impact graph**
([`impact-graph.md`](../impact-graph.md)): from the relation footers, tooling can propagate a
refutation to its dependents (mark them *in question*), promote dependents when a gap closes, and
derive both a **status report** (what is proved/open/axiomatised/replicated) and, optionally, a
**SemVer-style release version** for a knowledge artifact (a paper or a formalized library):
breaking knowledge change → MAJOR, additive result/replication → MINOR, repair/cleanup → PATCH.
